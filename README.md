这是一个心理健康支持 agent 原型项目，包含项目内 Codex skills、安全规则和合成测试用例。

第一阶段聚焦焦虑支持。项目故意先从安全门开始，再逐步加入焦虑相关工作流。

## 当前状态

本项目目前是**安全与临床依据优先的原型仓库**，还不是完整可用的心理健康产品。

现阶段重点是建立：

- 非诊断边界；
- 危机与医疗红旗分流；
- 临床资料优先的 skill 设计规范；
- 合成安全测试用例；
- 后续焦虑支持工作流的工程基础。

## 项目亮点

### 1. 临床资料优先，而不是模型自由发挥

项目的核心原则是 **evidence-first**：心理健康相关 skill 必须参考临床指南、公共健康资料、验证过的量表实践或明确标注为低风险的自助支持技术。

这意味着项目不会让模型凭空给出诊断、治疗建议或看似专业但没有来源的心理解释。

### 2. 不做诊断，不暴露疑似疾病标签

内部 formulation 只用于安全路由和选择支持路径，不应作为诊断或疾病标签展示给用户。

用户侧语言会优先表达为：

- “这不是诊断。”
- “这个模式值得和专业人士讨论。”
- “我们可以先做安全检查，再决定适合的下一步。”

而不是：

- “你有某种障碍。”
- “你可能是某个诊断。”
- “这个分数确认了某种疾病。”

### 3. 先安全分流，再自助支持

任何心理健康 workflow 都应先经过 `mental-health-safety-harness`。

如果出现危机、医疗红旗、药物或物质使用风险、严重功能受损等信号，应先进入安全分流，而不是继续普通焦虑练习。

### 4. 参考的临床资料方向

项目目前的设计参考方向包括：

- NICE CG113：成人广泛性焦虑和惊恐障碍管理；
- NICE NG225：自伤评估、管理和复发预防；
- NHS Talking Therapies：阶梯式心理治疗、标准化量表和 outcome monitoring；
- WHO 心理自助和可扩展心理干预相关资料；
- SAFE-T / C-SSRS 相关概念：自杀风险相关的直接询问和安全要素。

这些资料不会被用来让 agent 做诊断，而是用来约束：什么能做、什么不能做、什么时候需要转介真人或医疗支持。

## 和直接在 Codex 里对话有什么区别？

直接在 Codex 或通用聊天模型里对话，主要依赖当次 prompt、模型默认行为和用户临时提醒。这个 repo 的目标不是让模型“更会聊天”，而是把心理健康场景需要的**安全边界、临床依据和工作流约束工程化**。

主要区别：

| 维度 | 直接对话 | 本项目 |
|---|---|---|
| 临床依据 | 依赖模型临时回答，来源不一定可追踪 | 每个心理健康 skill 都应说明 clinical basis、适用范围和限制 |
| 安全顺序 | 可能直接进入安慰、分析或建议 | 先经过 safety harness，再决定是否可以继续普通支持 |
| 诊断边界 | 容易被用户诱导成“像医生一样判断” | 明确禁止诊断和疑似疾病标签外显 |
| 医疗红旗 | 可能被解释成焦虑体验 | 先按医疗风险处理，不把红旗症状说成“只是焦虑” |
| 可复用性 | 每次对话重新组织 | 规则、skills、docs、evals 都保存在 repo，可复审、复用、迭代 |
| 回归测试 | 很难知道改动是否破坏安全边界 | 通过合成 eval cases 和校验脚本逐步防止安全回退 |
| 扩展方式 | 继续加 prompt | 按 evidence policy 增加新的 skills 和 reference notes |

简言之：**直接对话是一次性的模型互动；本项目是一个可审查、可测试、可扩展的心理健康 agent 工程骨架。**

## 项目结构

- `.agents/skills/`：仅在本项目内使用的 Codex skills。
- `safety/`：共享的临床边界、安全分流和响应规则。
- `evals/`：合成安全测试和回归测试用例。
- `docs/`：设计说明和资料来源摘要。
- `scripts/`：本地检查和测试脚本。

## 当前 Skills

- `mental-health-safety-harness`：所有心理健康支持流程的上层安全门。
- `risk-crisis-triage`：危机、自伤、自杀、伤人、精神病性症状、严重功能受损、药物和医疗红旗分流。
- `anxiety-intake-assessment`：焦虑相关支持的非诊断结构化 intake。

## 非临床定位

本项目用于症状缓解、心理教育、自助指导、筛查辅助和转介支持。它不是诊断系统、治疗系统、药物建议系统或危机服务。

内部分类只用于选择更安全的支持路径，不应作为诊断或疾病标签展示给用户。
